    fe80::/10

  2. 组播地址

    ff00::/10

  3. 任播地址

    地址目前还在用全球可聚合单播地址，类似于组播地址，选择最近的地址发送，只发送一个地方。

  3. 嵌入式 X:X:X:X:X:X:d.d.d.d ::FFFF:192.168.0.1

CCIE 重分发

重分发

将不同路由协议转化

1直连进路由选择协议

redistribute connect metric 1

2其他路由选择协议进rip

router rip

router下 redistribute eigrp 100 metric 1值(默认0)

3其他路由选择协议进eigrp

router eigrp 100

redistribute rip metric 10000带宽 100延迟 255可靠性 1负载 1500MTU值（默认为以上数字，一般不改）

4其他进ospf

router os 1

redistribute eigrp 100进程号 subnets（子集，不加则为主类网络）

默认OE2

将oe2改成oe1:

re ei 100 subnets metric-type 1 (+metric 值）

重分发流量控制

acl

奇acc 1 permit 172.16.1.0 0.0.254.255

偶acc 1 permit 172.16.0.0 0.0.254.255

将acl抓去到的流量给route-map

route-map +名字 +permit/deny

优先级10(默认优先级)

match ip address +acl编号

route-map也默认deny所有流量

如需放行所有流量 route-map 名字 permit +下一个优先级20(一般为10的叠加)

re rip router-map ccie metric ……

CCIE BGP 补充知识

BGP 的路由更新过滤

route-map、prefix-list、filter-list（筛选列表）

利用 route-map

可以是 in 方向，也可以是 out 方向。

route-map 有隐藏的 deny all。

利用 prefix-list

route-map 不光可以完成过滤，还可以对放行的网络设置策略。

prefix-list 只能完成过滤。

prefix-list 也有默认的 deny all。

prefix-list 的放行所有 ip prefix-list x permit 0.0.0.0/0

利用 正则表达式

正则表达式匹配击中的是 AS 路径信息。

放行所有：ip as-path <id> permit _

BGP 的公认团体属性限制路由更新的发送

路由更新携带 no-advertise，路由更新被接收后不发送给任何 BGP peer。

路由更新携带 no-export 或 local-as，路由更新被接收后不发送给任何 eBGP peer。

查看BGP转发表中携带特定团体属性的路由更新show ip bgp community <no-export/no-advertise>

CCIE DM-VPN

DM-VPN

动态多点VPN、Cisco私有

IPSEC VPN 存在的问题

  1. 不支持多播
  2. 复杂的拓扑

DM-VPN 的术语

  • M-GRE（多点隧道）一点对多点
  • 物理 IP 地址（物理接口）/逻辑 IP 地址（tunnel 私有 IP）
  • NHRP（下一跳解析协议、总部和分部都启用）
    NHS（下一跳服务器、总部）

DM-VPN 的三个阶段

阶段一

  • M-GRE：EIGRP
  • 分支和分支要通过总部才能相互通讯

阶段二

  • M-GRE：EIGRP
  • 分支和分支可以直接通讯

阶段三

  • 单云双中心（两个 HUB 同时工作互为备份，还是依赖 DM-VPN 阶段2）

DM-VPN 的基础配置流程

步骤1：完成 HUB 上 M-GRE 的配置

步骤2：完成 spoke 上的 M-GRE 配置

步骤3：完成 HUB 上隧道 NHRP 的配置

步骤4：完成 spoke 上 NHRP 的配置

步骤5：隧道逻辑 IP 地址参与 EIGRP

步骤6：完成 IPSEC 的配置并且套用到隧道上

杂：

查看 HUB 的分支信息：show ip nhrp

关闭 EIGRP 水平分割：no ip split-horizon eigrp 1

开启 NHRP 捷径：ip nhrp shortcat

单云双中心的配置细节

区别1：两个 HUB 需要互做 NHRP 的映射。

区别2：两个 spoke 需要设定两个 NHS 服务器，并且谁先被设定谁就是主设备。

CCIE 跨域的多播

跨域的多播

源和组成员在不同的AS，需要传递多播数据流。

跨域多播的问题和挑战

组成员无法发现源、源无法发现组成员、多播树无法建立。

MSDP 多播源发现协议

MSDP 是配置在不同 AS RP 上的，它会建立基于 TCP 的对等体关系，利用自身的 SA（源关联）消息共享”源地址信息、多播组地址信息、RP 信息”给对端 RP。

MSDP 配置流程

步骤1：RP上配置新的 loopback 接口，并且宣告进 BGP。

步骤2：RP上完成 MSDP 的配置

排错手段

ip 路由疑难解析

多播排错

ping 多播组地址 不同

  1. 确认发现组成员

    show ip igmp group

    查看组地址、本地接口、报告者 是否出现

    错点

    • 没有模拟成组成员，缺乏接口下：ip igmp join-group <a.b.c.d>
    • 路由器接口没有加入PIM，缺乏接口下:ip pim <x>
    • 连通性问题

  2. 确认稀疏模式获取 RP 信息

    show ip pim rp mapping 查看 RP 映射，查看 RP 的结果

    show ip pim rp 查看多播组和 RP 的关联

    错点

    • 没有通过静态 RP 指明 RP 地址
    • BSR 的 C-BSR 或 C-RP 配置问题
    • auto-rp 的配置问题
    • 缺乏auto-rp 侦听的配置
    • 设备间缺乏必要的PIM 邻居：show ip pim neighbor
      • 设备缺乏多播路由功能和接口忘记加入PIM
      • 接口忘记激活、错误的安全策略挡住了PIM消息
    • 设备无法正常抵达 RP 地址，所以不显示获取的 RP 消息
      • 路由层面问题、忘记宣告、忘记重分发、错误的过滤

  3. RPF 检测未通过

    • 缺乏必要的更新源路由
    • 忘记宣告、忘记重分发、错误的路由更新过滤

  4. 是否是跨域多播、缺乏必要的配置和连通性

    • 缺乏必要的 MSDP 配置和 MSDP 对等体关系

      show run | sec ip msdp

      show ip msdp summary

      MSDP peer 地址不可达

      MSDP 链接源和起源 ID 设置错误

    • RPF 检测不通过

  5. 错误的安全策略

结束

CCIE STP生成树协议

STP生成树协议 7.27

一个广播域中选举一个生成树。

广播风暴：绝不停止。

重复帧。

mac地址表的不稳定。

1一个网络中选取一个根桥

2根桥上所有端口都为指定端口

3非根桥上选取一个根端口

4每一段选举一个指定端口

5非指定端口为阻塞端口

*粗体为生成树定义的交换机接口角色

根端口：距离根桥距离最短的端口。开销值一样的话就选最小的发送者id。只在非桥根上选举，并且只选一个。

指定端口：可以执行数据的转发。一台交换机上可以有多个指定端口。根桥上所有端口都是指定端口。

端口状态

Blocking（阻塞状态）：此时，二层端口为非指定端口，也不会参与数据帧的转发。该端口通过接收BPDU来判断根交换机的位置和根ID，以及在STP拓扑收敛结束之后，各交换机端口应该处于什么状态，在默认情况下，端口会在这种状态下停留20秒钟时间。

Listening（侦听状态）：生成树此时已经根据交换机所接收到的BPDU而判断出了这个端口应该参与数据帧的转发。于是交换机端口就将不再满足于接收BPDU，而同时也开始发送自己的BPDU，并以此通告邻接的交换机该端口会在活动拓扑中参与转发数据帧的工作。在默认情况下，该端口会在这种状态下停留15秒钟的时间。

Learning(学习状态)：这个二层端口准备参与数据帧的转发，并开始填写MAC表。在默认情况下，端口会在这种状态下停留15秒钟时间。

Forwarding（转发状态）：这个二层端口已经成为了活动拓扑的一个组成部分，它会转发数据帧，并同时收发BPDU。

Disabled（禁用状态）：这个二层端口不会参与生成树，也不会转发。

CCIE IPSEC VPN

IPSEC VPN

Ip security vpn，是一个工作在网络层的 VPN 技术。

数据在公共网络上面临的风险

  1. 不加密，明文传递
  2. 没有完整性保护，数据容易被篡改
  3. 没有对等体认证，容易受到中间人攻击
  4. 抵赖

IPSEC VPN 解决的问题

  1. 对等体认证，数据发送之前必须经过认证且通过

    口令认证、数字证书（计算机世界的身份证）、加密随机数

  2. 数据加密

    des、3des、aes

  3. 数据完整性保护

    HMAC：哈希消息认证码

  4. 数字签名，防抵赖

    DSA（RSA 公司开发）

KEY：密匙

KEY 是一个工具，可以参与算法作为变量出现，也可以参与认证作为口令出现。

对称密钥、非对称密钥

（加密算法中的 KEY 的理解）

  1. 对等密钥及对称密钥加密算法

    加密解密密钥是同一把

    坏处：不够安全、Key 的共享存在风险

    好处：简单高效

  2. 非对称密钥及非对称密钥算法

    非对称密钥是一对KEY，公钥和密钥。

    公钥最后会公开，密钥不公开。

    公钥加密的数据，只有私钥可以解开。

    私钥加密的数据，只有公钥可以解开。

    坏处：复杂，数据会变庞大

    好处：安全可靠

IPSEC VPN 的加密密钥

非对称密钥共享是用对称密钥及算法保护

数据本身是用非对称密钥及算法保护

密钥产生过程和共享方法

DH 算法负责产生密钥和共享密钥

IPSEC VPN 的关键组件

ISAKMP：互联网安全关联密钥管理协议

  • ISA：通讯双方建立了安全连接就叫做一个 SA 成功了。

    IPSEC VPN 的 SA 有两个：一个是保护 VPN 的，一个是保护数据的。

  • KMP：KMP 是通过 IKE（互联网密钥交换机制）完成的，IKE 专门负责产生密钥，共享密钥。

    IKE 分两个阶段：阶段1产生和共享VPN自身所需的密钥，阶段2利用阶段1的密钥算出新密钥，用于保护用户数据。

ESP 和 AH

ESP 和 AH 都是 VPN 头部信息的缩写

帧头、包头、段头、数据、帧尾

数据经历 VPN 保护后变化成如下：

帧头、VPN 头、包头、段头、数据、帧尾

VPN 头就分两种：

  1. ESP：封装安全静荷

    该头负责保护数据的完整性，同时加密数据

  2. AH：认证头

    该头只负责保护数据完整性，不加密数据

IPSEC VPN 的模式

  1. 隧道模式（tunnel）

    VPN 网关通过一个隧道模式的 VPN 连接，保护身后所有的通讯数据。

  2. 传输模式（transport）

    通讯双方自身建立 VPN 连接，保护通讯数据。

区别：

隧道模式的新包头，目的 IP 和源 IP 是 VPN 的对等体地址。

传输模式的新包头，目的 IP 和源 IP 复制了原有的 IP 地址。

NAT 对 IPSEC VPN 的影响

VPN 流量几乎永远是先被 inside 接口接收，所以 NAT 会直接修改 IP 包头，导致 VPN 无法识别感兴趣流。
