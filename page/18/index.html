CCIE DM-VPN

DM-VPN

动态多点VPN、Cisco私有

IPSEC VPN 存在的问题

  1. 不支持多播
  2. 复杂的拓扑

DM-VPN 的术语

  • M-GRE（多点隧道）一点对多点
  • 物理 IP 地址（物理接口）/逻辑 IP 地址（tunnel 私有 IP）
  • NHRP（下一跳解析协议、总部和分部都启用）
    NHS（下一跳服务器、总部）

DM-VPN 的三个阶段

阶段一

  • M-GRE：EIGRP
  • 分支和分支要通过总部才能相互通讯

阶段二

  • M-GRE：EIGRP
  • 分支和分支可以直接通讯

阶段三

  • 单云双中心（两个 HUB 同时工作互为备份，还是依赖 DM-VPN 阶段2）

DM-VPN 的基础配置流程

步骤1：完成 HUB 上 M-GRE 的配置

步骤2：完成 spoke 上的 M-GRE 配置

步骤3：完成 HUB 上隧道 NHRP 的配置

步骤4：完成 spoke 上 NHRP 的配置

步骤5：隧道逻辑 IP 地址参与 EIGRP

步骤6：完成 IPSEC 的配置并且套用到隧道上

杂：

查看 HUB 的分支信息：show ip nhrp

关闭 EIGRP 水平分割：no ip split-horizon eigrp 1

开启 NHRP 捷径：ip nhrp shortcat

单云双中心的配置细节

区别1：两个 HUB 需要互做 NHRP 的映射。

区别2：两个 spoke 需要设定两个 NHS 服务器，并且谁先被设定谁就是主设备。

路由器数据转发机制

CEF

FIB 转发信息库（本身就是路由表的副本）

RIB 路由表（给FIB提供要素）

MPLS详解

multiple protocol label switch（多协议标签交换）

MPLS作用

MPLS是一种数据交换机制，基于标签库和数据携带的标签。

进行数据转发，同时支持多种协议。

MPLS在实际工作中，能实现如下两个基本功能：

  1. 数据包的高速转发
  2. 减少参与BGP的设备数量、减少对等体关系、减少不必要的路由学习

MPLS术语

label 标签

总长 32bit 由如下几个部分组成：

前20bit 是标签取值范围

第21bit 是栈底位，栈底位为1则表明该标签已经是最后一个

第22~24bit 是体验位，可以将标签对应的数据分成8个级别，给予不同的传输策略。

最后8bit 是TTL，限制携带标签的数据传递范围。

MPLS 模式

帧模式：frame mode（主流模式），标签被插在帧头的后面，包头前面。

信令模式：cell mode 标签被插在信令头后面。

LSR：标签交换路由器

启用了MPLS的路由器统称LSR。

入站 LSR （inbound LSR）：接收到 IP 数据包，压入标签，传递给 MPLS 网络。

链路中 LSR （link LSR）：接收到携带标签的数据，基于标签进行数据转发。

出站 LSR （outbound LSR）：负责将标签移除，并且变回 IP 数据报文。

压入、交换、移除、弹出

压入：入站LSR将标签压入数据。

交换：链路中LSR基于标签库，进行标签的置换、数据的传递。

移除：出站LSR将数据携带的所有标签全部删除。

弹出：出站LSR将数据携带的栈顶标签弹出，其他标签保留。

LSP：链路标签路径

是MPLS设备传递数据所需的路径。（即路由）

LSP是通过标签的关联产生的。

标签分配、分发、关联、维护协议

TDP：标记分发协议、思科私有

LDP：标签分发协议、公有公开

LDP详解

LDP作用

  1. 利用 hello 消息建立 LDP 邻接关系。
  2. 基于CEF给本地网络分配本地标签。
  3. 将网络对应的标签分发共享给邻居
  4. 基于底层协议完成本地标签与远程共享标签的关联，最终形成LSP。
  5. 负责监控LSP的有效性，并且维护它。

LDP邻接关系的建立

LDP 邻居的发现，通过hello消息，基于UDP，目标端口646。

LDP 利用TCP，目标端口646完成邻接关系建立。

LDP可以设定RID，RID扮演更新源，更新源会成为对端TCP的目标接收地址，所以更新源必须可达。

LDP标签的分配

除BGP路由，LDP可以给本地直连网络以及非直连网络分配标签。

LDP标签的共享

两种形式：下游主动、上游主动。

实际上标签的分享是给所有的邻居，不遵循所谓的上下游。

LDP标签的关联

LDP永远将路由路径下一跳共享的标签与对应网络的本地标签关联。

boundaddress：绑定地址，参与MPLS设备的IP会成为绑定地址和，会跟着本地标签一起共享给LDP邻居，邻居根据底层协议下一跳地址和绑定地址的判断来确认哪个LDP邻居是下一跳，再绑定。

LDP的特殊标签

  • UNTAGE：未标记

出现原因：底层是ospf,本地网络无法找到对应的远程标签，LDP给网络分配untage。

处理方式：目标网络远程标签如果是untage，那么LSR会将所有标签移除，向外传递。

  • POP：倒数第二跳标签

出现原因：当前路由器给本地直连网络分配的本地标签就是POP。

处理方式：如果出站标签是POP，那么LSR 会将栈顶标签弹出。

意义：提高数据转发效率，减少一次标签库查找。

LDP的两张表

LIB：标签信息库

目标网络本地标签和学习到的远程标签存储在这里。（未关联）

LFIB：标签转发信息库

标签关联后的结果 即 LSP 存储在这里。

步骤5：接口下启用MPLS

mpls ip

2：IS-IS的发展历程

同时进行的，没有谁早谁晚的说法。

OSI七层模型 传输层 面向链接TCP 无链接UDP

OSI中后期产物 计算机网络最早的发展阶段提出的是CLNS和CONS

CLNS和CONS算是OSI的前身

N代表网络，S代表服务，C代表链接 O代表Of“有链接的网络服务”类似于TCP L代表Less “无链接的网络服务”

基于对CONS的研究，延伸出了IP【网络标识】和TCP

TCP数据传输控制 IP网络标识

因此开发出了动态的路由协议：RIP EIGRP OSPF BGP

另一阵营：CLNS坚持使用无链接的网络协议

开发出了CLNP：CLNS网络标识

还需要的机制：CLNS【名字重复】IS-IS

OSPF为IP服务，IS-IS为CLNP网络服务。

这些都是不同组织在同时发展的，最后CONS阵营的方案得到公司的认可。

最后，ISO组织提出了OSI七层模型，但是CLNS的理念也得到认可，因此保留下来，叫做UDP

最后，IS-IS就感觉要消失了

这是为了统一行业标准。

OSPF虚链路：①克服孤岛区域：0，1，2区域不互相学路由

②两边区域0，中间区域1，但是两边区域0不互相学路由。

IS-IS没有这个问题。

外企推运营商用IS-IS 思科排第一

割接：干掉原有协议，换新协议

两种方法：①升级OSPF，使其拥有IS-IS特性

②直接用IS-IS取代OSPF

两种方法都不太科学……

所以我们现在使用的IS-IS，并不是纯IS-IS

集成IS-IS、双栈IS-IS

所以很多技术原来与IP对立，后来被IP吞并……

15：ISIS 认证 及 ISIS 路由泄露

15.1）ISIS 认证

  1. 链路认证 isis password cisco

边界网关路由协议

BGP简介

工作类型：路径矢量路由选择协议

基于tcp(三次握手)

三张表：邻居表（peer表） bgp转发表 路由表

BGP版本

BGPv4：IPv4

BGPv4+：IPv6

MP-BGP：协议npls

M-BGP：多播

BGP优缺点

优点

1EGP路由表只能存储千条信息，bgp可上万条。

2丰富的路径属性，13条选路原则，提供了更丰富的路由操控策略。

缺点

1对路由器性能要求较高。

2配置程序比较复杂。

3一般小网络不建议使用bgp

AS 和 邻居类型

bgp AS 一台路由器只能属于一个

公网AS1-65411 65412-65535私有AS

iBGP：处于同一AS内的BGP路由器所建立的BGP对等体关系

eBGP：不同AS内的BGP路由器建立的…

BGP消息

open：建立成功，会互相发送open信息(类似hello)协商邻居关系的各项参数，等待对方确认。

keeplive：维持对等体(类似邻居)关系的建立，60s发一次，180s后peer关系结束

update：不会周期性更新，只会增量更新或触发更新。

Notification：当BGP检测到错误状态时，就向对等体发出Notification消息，之后BGP连接会立即中断。

peer关系建立过程

IDLE（空闲状态）：本地路由向外发送tcp(端口179)请求，等回复

connect状态：tcp建立

active：建立不成功，再次请求tcp连接的建立（回到IDLE）。

open sent：建立成功，发送open消息

openconfirm：收到对方的信息进行参数对比，参数一致则建立完成。

established：连接建立。。update。。keepalive。。

bgp的防环机制

iBGP-水平分隔原则

BGP对等体路由器收到iBGP路由，不会发送给其他iBGP对等体路由器。

但是纯水平拓扑上该原则会导致路由不被传递学习。两个解决方案

全互联拓扑解决方案：所有路由器间都建立iBGP关系。（垃圾方案）

路由反射器解决方案：

eBGP

基于AS PATH信息的检测。接收到来自于eBGP peer的路由更新，更新所带AS-PATH 信息包含了当前设备AS号，则该路由更新忽略。

特殊拓扑下路由反射器的防环机制

originate id：起源id。路由器将设备的RID设置为起源ID。起源ID和当前设备RID一样，则路由更新不学习。

cluster id：簇id。管理员在路由反射器上设置，路由更新携带的簇ID一致，则路由更新不学习。

同步与下一跳判断

路由更新来自于iBGP peer时才考虑同步和下一跳。

同步机制

对于当前启用BGP的设备，路由更新对应的网络底层协议也必须能学习到，这才叫同步。同步是为了避免路由黑洞。

现实中同步被关闭，路由黑洞问题无法避免，但利用MPLS能解决。

bgp路由黑洞

路由更新的传递设备也就是数据传输的沿途设备，并没有参与BGP ，学习该路由，所以数据抵达后，只能丢包。

解决方法：

1重分发(不建议，会增大路由表)

2全互联的网络拓扑(两两之间建立ibgp对等体)

下一跳判断

来自于iBGP peer的路由更新，如果下一跳不可达，路由更新不学习。

路由源于AS外，通告者是eBGP peer，下一跳就是eBGP peer地址。

路由源于AS外，通告者是iBGP peer，下一跳依旧是eBGP peer地址。此时下一跳可能不可达。

路由源于iBGP peer，通告者是iBGP peer，下一跳就是iBGP peer地址。

    ​ AA:NN

    ​ AS号：标记值

BGP选路原则

选路原则决定了BGP按什么标准进行路由更新属性的比较

  1. 优选拥有最高权重值的路径 weight

    用于影响当前路由器出站路径选择

    默认32768，越大越优

    不同类型的对等体出现，需要影响路径抉择，权重更优。

  2. 优选LP大的

  3. 优选起源于本地的路径

    next-hop=0.0.0.0

  4. 优选拥有最短AS-PATH 路径信息的

  5. 比较Origin属性<IGP优先于EGP优先于Incomplete>

  6. 优选拥有最佳MED值的

  7. eBGP peer 的路径优于 iBGP peer的路径

  8. 优选离下一跳更近的路径

  9. 针对eBGP peer的路由，通过命令行允许多条路径，成为最佳进入路由表

  10. 优选最早被学习的路由，只针对 eBGP peer

  11. 优选拥有最小RID的BGP peer提供的路由

  12. 优选拥有最短cluster-id 的路径

  13. 优选拥有最低peer地址的邻居提供的路径

JSON

JavaScript 对象表示法

JSON 是存储和交换文本信息的语法。类似 XML。

JSON 比 XML 更小、更快，更易解析。

JSON 使用 JavaScript 语法来描述数据对象，但是 JSON 仍然独立于语言和平台。JSON 解析器和 JSON 库支持许多不同的编程语言。

JSON - 转换为 JavaScript 对象

JSON 文本格式在语法上与创建 JavaScript 对象的代码相同。

由于这种相似性，无需解析器，JavaScript 程序能够使用内建的 eval() 函数，用 JSON 数据来生成原生的 JavaScript 对象。

为什么使用 JSON？

对于 AJAX 应用程序来说，JSON 比 XML 更快更易使用：

  1. 使用 XML
  • 读取 XML 文档
  • 使用 XML DOM 来循环遍历文档
  • 读取值并存储在变量中
  1. 使用 JSON
  • 读取 JSON 字符串
  • 用 eval() 处理 JSON 字符串

JSON 语法规则

JSON 语法是 JavaScript 对象表示法语法的子集。

  • 数据在名称/值对中
  • 数据由逗号分隔
  • 花括号保存对象
  • 方括号保存数组

JSON 值可以是：

  • 数字（整数或浮点数）
  • 字符串（在双引号中）
  • 逻辑值（true 或 false）
  • 数组（在方括号中）
  • 对象（在花括号中）
  • null

JSON 解析器

提示：eval() 函数可编译并执行任何 JavaScript 代码。这隐藏了一个潜在的安全问题。

使用 JSON 解析器将 JSON 转换为 JavaScript 对象是更安全的做法。JSON 解析器只能识别 JSON 文本，而不会编译脚本。

在浏览器中，这提供了原生的 JSON 支持，而且 JSON 解析器的速度更快。

较新的浏览器和最新的 ECMAScript (JavaScript) 标准中均包含了原生的对 JSON 的支持。

mysql> grant all privileges on *.* to 'skuaka'@'%' identified by '你想设置的密码' with grant option;

  • all privileges：表示将所有权限授予给用户。也可指定具体的权限，如：SELECT、CREATE、DROP等。
  • on：表示这些权限对哪些数据库和表生效，格式：数据库名.表名，这里写*.*表示所有数据库，所有表。如果我要指定将权限应用到test库的user表中，可以这么写：test.user
  • to：将权限授予哪个用户。格式：”用户名”@”登录IP或域名”。%表示没有限制，在任何主机都可以登录。比如：”skuaka”@”192.168.0.%”，表示yangxin这个用户只能在192.168.0IP段登录。
  • identified by：指定用户的登录密码
  • with grant option：表示允许用户将自己的权限授权给其它用户
