CCIE IPSEC VPN

IPSEC VPN

Ip security vpn，是一个工作在网络层的 VPN 技术。

数据在公共网络上面临的风险

  1. 不加密，明文传递
  2. 没有完整性保护，数据容易被篡改
  3. 没有对等体认证，容易受到中间人攻击
  4. 抵赖

IPSEC VPN 解决的问题

  1. 对等体认证，数据发送之前必须经过认证且通过

    口令认证、数字证书（计算机世界的身份证）、加密随机数

  2. 数据加密

    des、3des、aes

  3. 数据完整性保护

    HMAC：哈希消息认证码

  4. 数字签名，防抵赖

    DSA（RSA 公司开发）

KEY：密匙

KEY 是一个工具，可以参与算法作为变量出现，也可以参与认证作为口令出现。

对称密钥、非对称密钥

（加密算法中的 KEY 的理解）

  1. 对等密钥及对称密钥加密算法

    加密解密密钥是同一把

    坏处：不够安全、Key 的共享存在风险

    好处：简单高效

  2. 非对称密钥及非对称密钥算法

    非对称密钥是一对KEY，公钥和密钥。

    公钥最后会公开，密钥不公开。

    公钥加密的数据，只有私钥可以解开。

    私钥加密的数据，只有公钥可以解开。

    坏处：复杂，数据会变庞大

    好处：安全可靠

IPSEC VPN 的加密密钥

非对称密钥共享是用对称密钥及算法保护

数据本身是用非对称密钥及算法保护

密钥产生过程和共享方法

DH 算法负责产生密钥和共享密钥

IPSEC VPN 的关键组件

ISAKMP：互联网安全关联密钥管理协议

  • ISA：通讯双方建立了安全连接就叫做一个 SA 成功了。

    IPSEC VPN 的 SA 有两个：一个是保护 VPN 的，一个是保护数据的。

  • KMP：KMP 是通过 IKE（互联网密钥交换机制）完成的，IKE 专门负责产生密钥，共享密钥。

    IKE 分两个阶段：阶段1产生和共享VPN自身所需的密钥，阶段2利用阶段1的密钥算出新密钥，用于保护用户数据。

ESP 和 AH

ESP 和 AH 都是 VPN 头部信息的缩写

帧头、包头、段头、数据、帧尾

数据经历 VPN 保护后变化成如下：

帧头、VPN 头、包头、段头、数据、帧尾

VPN 头就分两种：

  1. ESP：封装安全静荷

    该头负责保护数据的完整性，同时加密数据

  2. AH：认证头

    该头只负责保护数据完整性，不加密数据

IPSEC VPN 的模式

  1. 隧道模式（tunnel）

    VPN 网关通过一个隧道模式的 VPN 连接，保护身后所有的通讯数据。

  2. 传输模式（transport）

    通讯双方自身建立 VPN 连接，保护通讯数据。

区别：

隧道模式的新包头，目的 IP 和源 IP 是 VPN 的对等体地址。

传输模式的新包头，目的 IP 和源 IP 复制了原有的 IP 地址。

CCIE 多播（multicast）

多播（multicast）

基础知识

数据的传递方式

  • 单播（unicast）

    一对一通讯，数据接收者已知。

  • 广播（boradcast）

    一对所有通讯，数据接收者未知。

  • 多播（mulicast）

    一对多通讯，数据接收者未知。

路由器和交换机对不同类型数据的处理方式

路由器

单播包：路由或丢弃

广播包：丢弃或本地存在感兴趣的进程进行处理

多播包：丢弃或本地存在感兴趣的进程进行处理

交换机

单播帧：目的MAC地址交换机是否已知，如果已知则交换，如果未知则泛洪

广播帧：泛洪

多播帧：泛洪

IGMP 互联网组管理协议

路由器及终端运行，帮助路由器发现直连子网中的多播接收者。

组（group）：所有对同一多播流感兴趣的成员组成了多播组。

组成员（group member）：组中的多播流接收者。

版本：IGMPv1、IGMPv2、IGMPv3

IGMPv1

  • 报告消息（report）：始发于终端，用于通告子网中的路由器存在特定组组成员。

    发送终端的IP 和MAC以及组信息。

  • 查询消息（request）：始发于路由器，用于查询直连子网中是否存在组成员。

    通常是路由器接收到多播流后触发。

一旦发现组成员，路由器就创造一个180s 的老化计时器，该计时器通过 report 消息进行维护。

IGMPv1的缺陷

  1. 响应时间慢

    组中没有成员时，路由器需要 180s 才能发现

  2. 存在多查询者

  3. 存在多转发者

IGMPv2

  • 报告消息（report）：和 v1 相同

  • 查询消息（request）：和 v1 相同

  • 离开消息（leave）：始发于终端，终端PC 不再是组成员时发送，用于通告路由器有组成员离开。

    收到离开消息时，立刻发送SQ，如下。

  • 特定查询消息（SQ）：始发于路由器，发送出去后设定的老化时间为 3s 。如果 3s 内收到 report ，说明发送离开消息的组成员并不是组中最后一个人；如果收不到任何消息，那么说明组成员不存在。

IGMPv2 查询者竞选机制：多个查询者通过比较查询消息源 ip 来确认谁最终负责查询，源 ip 越小越优。

多转发者缺陷 IGMPv2 无法克服。

多播地址

多播 IP 和多播 MAC 都不是用来定位接收者所在的网络和物理位置。

它们用于标识服务和应用。

RIPv2 224.0.0.9

IPv4 D类 224.x.x.x 239.x.x.x

多播 mac

设备根据多播的 IP 自动计算而来。

多播的MAC地址可能会重复，多播的 IP 后 23bit 一致则会重复。

多播路由器协议

是运行在路由器之间，负责建立多播路由路径，用于传递多播数据。

多播路由协议分类

  • 距离矢量：DVMRP 距离矢量多播路由协议（淘汰了）
  • 链路状态：MOSPF 多播的OSPF（非主流）
  • 距离向量：CBT 基于核心的树（非主流）
  • 独立于其他：PIM 协议独立多播（Cisco、主流）

PIM 的两个版本

PIMv1：用的是 IGMPv1，但是现在也用 IGMPv2

PIMv2：用的是 IGMPv2

PIM 的三种模式

  • 密集模式（dense mode）

    设备间带宽资源紧张，子网成员大部分是组成员的时候，建议使用密集模式。

    密集模式默认采用 IGMPv1。

  • 稀疏模式（sparse mode）

    设备间带宽资源依旧紧张，子网成员只有少数是组成员，建议使用稀疏模式。

    稀疏模式采用 IGMPv2。

  • 混合模式（sparse-dense mode）

    该模式下路由器先启用稀疏模式，稀疏模式失效后再用密集模式。

PIM 的密集模式详解

消息

  • 加入消息（join）：下游路由器通过加入消息将直连网络加入多播转发路径。
  • 修剪消息（prune）：下游路由器通过修剪消息，将原本隶属于多播路径的直连网络从多播路径中删除。
  • 接枝消息（graft）：下游路由器通过该消息请求将原本被拆除的多播路径再恢复过来。
  • 接枝确认消息（graft-ack）：上游路由器通过该消息告诉下游，直连路径恢复成多播转发路径。
  • assert消息（assert）：连接子网的路由器通过该消息，竞选出最终的转发者。

密集模式工作过程

扩散、剪枝。

PIM 的特殊机制

  • PIM 密集模式的 DR

    接口优先级+接口ip地址，构成竞选参数，越大越优。

    PIM密集模式早期使用IGMPv1，IGMPv1会出现多查询者问题，于是就利用DR的竞选，来帮助IGMPv1确认谁负责查询。

    PIM密集模式现在使用的是IGMPv2，DR就没用了。

  • PIM 消息中的 assert 作用

    通过竞选选出一个人负责转发数据。

    assert 消息，携带了三个竞选参数：

    • 当前路由器启用的底层协议管理距离，越小越优。
    • 当前路由器抵达多播源底层路由协议的开销，越小越优。
    • 当前路由器连接子网的接口 ip，越大越优。

PIM 密集模式的配置流程

  • 步骤1：确认参与多播的设备及接口

    所有需要接收多播流的接口和发送多播流的接口及连接组成员的接口，都要参与多播路由协议的启用。

  • 步骤2：路由器接口模拟成多播组成员

    注意：该命令可以反复录入，一个接口扮演多个组成员

  • 步骤3：路由器开启多播路由功能，接口启用多播密集模式

  • 步骤4：校验查看

    校验路由器是否发现了组成员：show ip igmp group

    校验路由器是否形成了PIM 邻居关系：show ip pim neighbor

    校验路由器多播路由表：show ip mroute

多播路由表

父条目与子条目

  • 父条目

    父条目起提醒注释的作用，告知当前路由器和管理员，路由器可能需要为某个多播组传递流量。

    路由器收到 report 消息便会产生父条目，路由器收到多播流也会产生父条目。

  • 子条目

    子条目是实际负责多播流量传递工作。

    子条目是路由器根据计算得来。

多播路由条目的样式

（多播源，组地址） 存活计时器/老化计时器

Incoming-interface（上游接口）：接口名字 状态/模式

Outgoing-interface（上游接口）：接口名字 状态/模式

父条目多播源部分几乎都是*号，代表源未知。

子条目多播源部分一定是某个 IPv4 单播地址。

多播防环机制

RPF：反向路径转发：启用多播的接口默认开启，用于避免环路的出现。

所有接收多播流的接口，必须通过 RPF 检测，才可以成为有效的上游接口。

当前接口必须是抵达多播源下一跳地址的本地出口，才通过检测。

用来查看rpf检测是否通过：show ip rpf <intf-id>

多播树

多播树其实就是多播转发数据的路径。

密集模式的树

最短路径树，也称源树。

当前路由器以多播源为根，计算抵达多播源的最短路径，每个路径对应到多播路由表中的一个条目。

源树缺陷
  • 容易产生大量的多播子条目。
  • 树构建的早期多播流被扩散到了不必要的设备上

PIM 的稀疏模式详解

消息

join、prune、graft、graft-ack、assert（和密集模式相同）

注册消息（register）

注册停止消息（register-stop）

稀疏模式中的集合点：RP

RP 是管理员指定的，是多股多播流的交汇处。

稀疏模式的工作原理

上游注册，下游加入，RP 整合。

！！注意

RP 以下的设备，以 RP 为根，计算最短路径。

RP 自身以源为根，计算最短路径。

稀疏模式是一个双树结构，RP 和源之间维护的是源树，下游设备与 RP 之间维护的是共享树。

共享树切换成源树

稀疏模式下，每台支持多播路由协议的路由器在接收超过4个字节的多播流后，就会尝试建立源树。

      RP-Agent：RP代理、C-RP 信息收集，裁决，分享结果

      RP：RP

    2. BSR：boot strap router：自举路由器

      C-BSR：BSR 候选人

      BSR：信息收集，信息分享

      C-RP、RP

      BSR 的竞选过程：BSR 标识越大越优。

      RP

      C-RP 有三个竞选参数：优先级（小优）、hash 长度（长优）、C-RP 标识（大优）

PIM 的混合模式

auto-rp 的问题和解决方案

RP-Agent 将结果发向 224.0.1.40

通告结果是一跳，如果拓扑是水平的多台设备组成，那么就会出现有设备无法得知 RP 信息。

网关冗余对多播传递的影响

MPLS详解

multiple protocol label switch（多协议标签交换）

MPLS作用

MPLS是一种数据交换机制，基于标签库和数据携带的标签。

进行数据转发，同时支持多种协议。

MPLS在实际工作中，能实现如下两个基本功能：

  1. 数据包的高速转发
  2. 减少参与BGP的设备数量、减少对等体关系、减少不必要的路由学习

MPLS术语

label 标签

总长 32bit 由如下几个部分组成：

前20bit 是标签取值范围

第21bit 是栈底位，栈底位为1则表明该标签已经是最后一个

第22~24bit 是体验位，可以将标签对应的数据分成8个级别，给予不同的传输策略。

最后8bit 是TTL，限制携带标签的数据传递范围。

MPLS 模式

帧模式：frame mode（主流模式），标签被插在帧头的后面，包头前面。

信令模式：cell mode 标签被插在信令头后面。

LSR：标签交换路由器

启用了MPLS的路由器统称LSR。

入站 LSR （inbound LSR）：接收到 IP 数据包，压入标签，传递给 MPLS 网络。

链路中 LSR （link LSR）：接收到携带标签的数据，基于标签进行数据转发。

出站 LSR （outbound LSR）：负责将标签移除，并且变回 IP 数据报文。

压入、交换、移除、弹出

压入：入站LSR将标签压入数据。

交换：链路中LSR基于标签库，进行标签的置换、数据的传递。

移除：出站LSR将数据携带的所有标签全部删除。

弹出：出站LSR将数据携带的栈顶标签弹出，其他标签保留。

LSP：链路标签路径

是MPLS设备传递数据所需的路径。（即路由）

LSP是通过标签的关联产生的。

标签分配、分发、关联、维护协议

TDP：标记分发协议、思科私有

LDP：标签分发协议、公有公开

LDP详解

LDP作用

  1. 利用 hello 消息建立 LDP 邻接关系。
  2. 基于CEF给本地网络分配本地标签。
  3. 将网络对应的标签分发共享给邻居
  4. 基于底层协议完成本地标签与远程共享标签的关联，最终形成LSP。
  5. 负责监控LSP的有效性，并且维护它。

LDP邻接关系的建立

LDP 邻居的发现，通过hello消息，基于UDP，目标端口646。

LDP 利用TCP，目标端口646完成邻接关系建立。

LDP可以设定RID，RID扮演更新源，更新源会成为对端TCP的目标接收地址，所以更新源必须可达。

LDP标签的分配

除BGP路由，LDP可以给本地直连网络以及非直连网络分配标签。

LDP标签的共享

两种形式：下游主动、上游主动。

实际上标签的分享是给所有的邻居，不遵循所谓的上下游。

LDP标签的关联

LDP永远将路由路径下一跳共享的标签与对应网络的本地标签关联。

LDP的特殊标签

  • UNTAGE：未标记

出现原因：底层是ospf,本地网络无法找到对应的远程标签，LDP给网络分配untage。

  • POP：倒数第二跳标签

出现原因：当前路由器给本地直连网络分配的本地标签就是POP。

处理方式：如果出站标签是POP，那么LSR 会将栈顶标签弹出。

意义：提高数据转发效率，减少一次标签库查找。

originate id：起源id。路由器将设备的RID设置为起源ID。起源ID和当前设备RID一样，则路由更新不学习。

cluster id：簇id。管理员在路由反射器上设置，路由更新携带的簇ID一致，则路由更新不学习。

来自于iBGP peer的路由更新，如果下一跳不可达，路由更新不学习。

路由源于AS外，通告者是eBGP peer，下一跳就是eBGP peer地址。

路由源于AS外，通告者是iBGP peer，下一跳依旧是eBGP peer地址。此时下一跳可能不可达。

路由源于iBGP peer，通告者是iBGP peer，下一跳就是iBGP peer地址。
