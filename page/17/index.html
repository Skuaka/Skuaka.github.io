CCIE 交换机基础

交换机基础

泛洪：除接收端口外，所有端口都向外去发。

mac地址表=cam表

广播域：所能传播的最大范围

冲突域：发生冲突之后所影响的范围

CSMA/CD：载波帧听多链路访问

交换机指示灯：

绿色常亮：接口两端没有问题，但是当前没有数据从中通过

绿色不停地闪：正在传输数据

橘红色慢闪：端口没问题,对端线路或者对端的端口有问题

橘红色快闪：端口和对端连接没有问题，正在协商参数建立

黄色闪动：物理接口有问题，尝试重启。

不闪： 物理接口有问题，无法通信

vlan（虚拟局域网）

作用：分隔广播域

所有接口默认属于vlan1（默认vlan，不可删除）。

vlan ID范围和解释

\1. 0和4095 仅仅限于系统使用 用户不能查看和使用

\2. 1 正常 cisco默认的vlan,用户可以使用但是不能删除

\3. 2–1001 正常 用于以太网的VLAN，用户可以创建。使用。删除。

\4. 1002–1005 正常 用于FDDI和令牌环的CISCO默认VLAN，用户不能删除

\5. 1006–1024 保留 仅仅限于系统使用。用户不能查看和使用。

\6. 1025–4094 扩展 仅仅用于以太网VLAN

isl 思科私有协议，目前思科也不采用。通过再封装工作。

vtp（虚拟传输协议）

用于同步同一网络中交换机上网络的信息（减小庞大的配置量）

生效的前提：trunk正常工作

客户端：RAM(重启后要重新同步） 传递、同步vlan信息

CCIE 多播（multicast）

多播（multicast）

基础知识

数据的传递方式

  • 单播（unicast）

    一对一通讯，数据接收者已知。

  • 广播（boradcast）

    一对所有通讯，数据接收者未知。

  • 多播（mulicast）

    一对多通讯，数据接收者未知。

路由器和交换机对不同类型数据的处理方式

路由器

单播包：路由或丢弃

广播包：丢弃或本地存在感兴趣的进程进行处理

多播包：丢弃或本地存在感兴趣的进程进行处理

交换机

单播帧：目的MAC地址交换机是否已知，如果已知则交换，如果未知则泛洪

广播帧：泛洪

多播帧：泛洪

IGMP 互联网组管理协议

路由器及终端运行，帮助路由器发现直连子网中的多播接收者。

组（group）：所有对同一多播流感兴趣的成员组成了多播组。

组成员（group member）：组中的多播流接收者。

版本：IGMPv1、IGMPv2、IGMPv3

IGMPv1

  • 报告消息（report）：始发于终端，用于通告子网中的路由器存在特定组组成员。

    发送终端的IP 和MAC以及组信息。

  • 查询消息（request）：始发于路由器，用于查询直连子网中是否存在组成员。

    通常是路由器接收到多播流后触发。

一旦发现组成员，路由器就创造一个180s 的老化计时器，该计时器通过 report 消息进行维护。

IGMPv1的缺陷

  1. 响应时间慢

    组中没有成员时，路由器需要 180s 才能发现

  2. 存在多查询者

  3. 存在多转发者

IGMPv2

  • 报告消息（report）：和 v1 相同

  • 查询消息（request）：和 v1 相同

  • 离开消息（leave）：始发于终端，终端PC 不再是组成员时发送，用于通告路由器有组成员离开。

    收到离开消息时，立刻发送SQ，如下。

  • 特定查询消息（SQ）：始发于路由器，发送出去后设定的老化时间为 3s 。如果 3s 内收到 report ，说明发送离开消息的组成员并不是组中最后一个人；如果收不到任何消息，那么说明组成员不存在。

IGMPv2 查询者竞选机制：多个查询者通过比较查询消息源 ip 来确认谁最终负责查询，源 ip 越小越优。

多转发者缺陷 IGMPv2 无法克服。

多播地址

多播 IP 和多播 MAC 都不是用来定位接收者所在的网络和物理位置。

它们用于标识服务和应用。

RIPv2 224.0.0.9

IPv4 D类 224.x.x.x 239.x.x.x

多播 mac

设备根据多播的 IP 自动计算而来。

多播的MAC地址可能会重复，多播的 IP 后 23bit 一致则会重复。

多播路由器协议

是运行在路由器之间，负责建立多播路由路径，用于传递多播数据。

多播路由协议分类

  • 距离矢量：DVMRP 距离矢量多播路由协议（淘汰了）
  • 链路状态：MOSPF 多播的OSPF（非主流）
  • 距离向量：CBT 基于核心的树（非主流）
  • 独立于其他：PIM 协议独立多播（Cisco、主流）

PIM 的两个版本

PIMv1：用的是 IGMPv1，但是现在也用 IGMPv2

PIMv2：用的是 IGMPv2

PIM 的三种模式

  • 密集模式（dense mode）

    设备间带宽资源紧张，子网成员大部分是组成员的时候，建议使用密集模式。

    密集模式默认采用 IGMPv1。

  • 稀疏模式（sparse mode）

    设备间带宽资源依旧紧张，子网成员只有少数是组成员，建议使用稀疏模式。

    稀疏模式采用 IGMPv2。

  • 混合模式（sparse-dense mode）

    该模式下路由器先启用稀疏模式，稀疏模式失效后再用密集模式。

PIM 的密集模式详解

消息

  • 加入消息（join）：下游路由器通过加入消息将直连网络加入多播转发路径。
  • 修剪消息（prune）：下游路由器通过修剪消息，将原本隶属于多播路径的直连网络从多播路径中删除。
  • 接枝消息（graft）：下游路由器通过该消息请求将原本被拆除的多播路径再恢复过来。
  • 接枝确认消息（graft-ack）：上游路由器通过该消息告诉下游，直连路径恢复成多播转发路径。
  • assert消息（assert）：连接子网的路由器通过该消息，竞选出最终的转发者。

密集模式工作过程

扩散、剪枝。

PIM 的特殊机制

  • PIM 密集模式的 DR

    接口优先级+接口ip地址，构成竞选参数，越大越优。

    PIM密集模式早期使用IGMPv1，IGMPv1会出现多查询者问题，于是就利用DR的竞选，来帮助IGMPv1确认谁负责查询。

    PIM密集模式现在使用的是IGMPv2，DR就没用了。

  • PIM 消息中的 assert 作用

    通过竞选选出一个人负责转发数据。

    assert 消息，携带了三个竞选参数：

    • 当前路由器启用的底层协议管理距离，越小越优。
    • 当前路由器抵达多播源底层路由协议的开销，越小越优。
    • 当前路由器连接子网的接口 ip，越大越优。

PIM 密集模式的配置流程

  • 步骤1：确认参与多播的设备及接口

    所有需要接收多播流的接口和发送多播流的接口及连接组成员的接口，都要参与多播路由协议的启用。

  • 步骤2：路由器接口模拟成多播组成员

    注意：该命令可以反复录入，一个接口扮演多个组成员

  • 步骤3：路由器开启多播路由功能，接口启用多播密集模式

  • 步骤4：校验查看

    校验路由器是否发现了组成员：show ip igmp group

    校验路由器是否形成了PIM 邻居关系：show ip pim neighbor

    校验路由器多播路由表：show ip mroute

多播路由表

父条目与子条目

  • 父条目

    父条目起提醒注释的作用，告知当前路由器和管理员，路由器可能需要为某个多播组传递流量。

    路由器收到 report 消息便会产生父条目，路由器收到多播流也会产生父条目。

  • 子条目

    子条目是实际负责多播流量传递工作。

    子条目是路由器根据计算得来。

多播路由条目的样式

（多播源，组地址） 存活计时器/老化计时器

Incoming-interface（上游接口）：接口名字 状态/模式

Outgoing-interface（上游接口）：接口名字 状态/模式

父条目多播源部分几乎都是*号，代表源未知。

子条目多播源部分一定是某个 IPv4 单播地址。

多播防环机制

RPF：反向路径转发：启用多播的接口默认开启，用于避免环路的出现。

所有接收多播流的接口，必须通过 RPF 检测，才可以成为有效的上游接口。

当前接口必须是抵达多播源下一跳地址的本地出口，才通过检测。

用来查看rpf检测是否通过：show ip rpf <intf-id>

多播树

多播树其实就是多播转发数据的路径。

密集模式的树

最短路径树，也称源树。

当前路由器以多播源为根，计算抵达多播源的最短路径，每个路径对应到多播路由表中的一个条目。

源树缺陷
  • 容易产生大量的多播子条目。
  • 树构建的早期多播流被扩散到了不必要的设备上

PIM 的稀疏模式详解

消息

join、prune、graft、graft-ack、assert（和密集模式相同）

注册消息（register）

注册停止消息（register-stop）

稀疏模式中的集合点：RP

RP 是管理员指定的，是多股多播流的交汇处。

稀疏模式的工作原理

上游注册，下游加入，RP 整合。

！！注意

RP 以下的设备，以 RP 为根，计算最短路径。

RP 自身以源为根，计算最短路径。

稀疏模式是一个双树结构，RP 和源之间维护的是源树，下游设备与 RP 之间维护的是共享树。

共享树切换成源树

稀疏模式下，每台支持多播路由协议的路由器在接收超过4个字节的多播流后，就会尝试建立源树。

RP 的获取方式

  • 手动指定：ip pim rp-address <RP地址>

  • RP 的竞选

    1. 代理 rp（auto-rp）

      C-RP：RP候选者

      RP-Agent：RP代理、C-RP 信息收集，裁决，分享结果

      RP：RP

    2. BSR：boot strap router：自举路由器

      C-BSR：BSR 候选人

      BSR：信息收集，信息分享

      C-RP、RP

      BSR 的竞选过程：BSR 标识越大越优。

      RP

      C-RP 有三个竞选参数：优先级（小优）、hash 长度（长优）、C-RP 标识（大优）

PIM 的混合模式

让设备启用混合模式：if)#ip pim sparse-dense-mode

混合模式倒退回密集模式：一定是 RP 彻底无法正常工作

Router(config)#no ip pim dm-fallback //不允许混合模式倒退回密集模式

CCIE IPSEC VPN

IPSEC VPN

Ip security vpn，是一个工作在网络层的 VPN 技术。

数据在公共网络上面临的风险

  1. 不加密，明文传递
  2. 没有完整性保护，数据容易被篡改
  3. 没有对等体认证，容易受到中间人攻击
  4. 抵赖

IPSEC VPN 解决的问题

  1. 对等体认证，数据发送之前必须经过认证且通过

    口令认证、数字证书（计算机世界的身份证）、加密随机数

  2. 数据加密

    des、3des、aes

  3. 数据完整性保护

    HMAC：哈希消息认证码

  4. 数字签名，防抵赖

    DSA（RSA 公司开发）

KEY：密匙

KEY 是一个工具，可以参与算法作为变量出现，也可以参与认证作为口令出现。

对称密钥、非对称密钥

（加密算法中的 KEY 的理解）

  1. 对等密钥及对称密钥加密算法

    加密解密密钥是同一把

    坏处：不够安全、Key 的共享存在风险

    好处：简单高效

  2. 非对称密钥及非对称密钥算法

    非对称密钥是一对KEY，公钥和密钥。

    公钥最后会公开，密钥不公开。

    公钥加密的数据，只有私钥可以解开。

    私钥加密的数据，只有公钥可以解开。

    坏处：复杂，数据会变庞大

    好处：安全可靠

IPSEC VPN 的加密密钥

非对称密钥共享是用对称密钥及算法保护

数据本身是用非对称密钥及算法保护

密钥产生过程和共享方法

DH 算法负责产生密钥和共享密钥

IPSEC VPN 的关键组件

ISAKMP：互联网安全关联密钥管理协议

  • ISA：通讯双方建立了安全连接就叫做一个 SA 成功了。

    IPSEC VPN 的 SA 有两个：一个是保护 VPN 的，一个是保护数据的。

  • KMP：KMP 是通过 IKE（互联网密钥交换机制）完成的，IKE 专门负责产生密钥，共享密钥。

    IKE 分两个阶段：阶段1产生和共享VPN自身所需的密钥，阶段2利用阶段1的密钥算出新密钥，用于保护用户数据。

ESP 和 AH

ESP 和 AH 都是 VPN 头部信息的缩写

帧头、包头、段头、数据、帧尾

数据经历 VPN 保护后变化成如下：

帧头、VPN 头、包头、段头、数据、帧尾

VPN 头就分两种：

  1. ESP：封装安全静荷

    该头负责保护数据的完整性，同时加密数据

  2. AH：认证头

    该头只负责保护数据完整性，不加密数据

IPSEC VPN 的模式

  1. 隧道模式（tunnel）

    VPN 网关通过一个隧道模式的 VPN 连接，保护身后所有的通讯数据。

  2. 传输模式（transport）

    通讯双方自身建立 VPN 连接，保护通讯数据。

区别：

隧道模式的新包头，目的 IP 和源 IP 是 VPN 的对等体地址。

传输模式的新包头，目的 IP 和源 IP 复制了原有的 IP 地址。

NAT 对 IPSEC VPN 的影响

VPN 流量几乎永远是先被 inside 接口接收，所以 NAT 会直接修改 IP 包头，导致 VPN 无法识别感兴趣流。

gcc常用标签

对于C/C++编写的程序，从源代码到可执行文件，一般经过下面四个步骤：

  1. 预处理，产生.ii文件

    gcc -E hello.c > hello.i 只进行预处理，会默认输出到标准输出。

  2. 编译，产生汇编文件(.s文件)

    gcc -O1 -S hello.c -S 产生汇编代码(注意是大写！！) hello.s ；-O1是程序优化等级，O2优化比O1高，查看汇编代码时使用低优化等级有助于阅读学习，高版本gcc可以使用-Og

  3. 汇编，产生目标文件(.o或.obj文件)

    gcc -O1 -c hello.c 汇编，阻止编译器创建一个完整的程序，产生二进制目标代码文件hello.o。

  4. 链接,产生可执行文件(.out或.exe文件)

    gcc -o hello hello.c

JSON

JavaScript 对象表示法

JSON 是存储和交换文本信息的语法。类似 XML。

JSON 比 XML 更小、更快，更易解析。

JSON 使用 JavaScript 语法来描述数据对象，但是 JSON 仍然独立于语言和平台。JSON 解析器和 JSON 库支持许多不同的编程语言。

JSON - 转换为 JavaScript 对象

JSON 文本格式在语法上与创建 JavaScript 对象的代码相同。

由于这种相似性，无需解析器，JavaScript 程序能够使用内建的 eval() 函数，用 JSON 数据来生成原生的 JavaScript 对象。

为什么使用 JSON？

对于 AJAX 应用程序来说，JSON 比 XML 更快更易使用：

  1. 使用 XML
  • 读取 XML 文档
  • 使用 XML DOM 来循环遍历文档
  • 读取值并存储在变量中
  1. 使用 JSON
  • 读取 JSON 字符串
  • 用 eval() 处理 JSON 字符串

JSON 语法规则

JSON 语法是 JavaScript 对象表示法语法的子集。

  • 数据在名称/值对中
  • 数据由逗号分隔
  • 花括号保存对象
  • 方括号保存数组

JSON 值可以是：

  • 数字（整数或浮点数）
  • 字符串（在双引号中）
  • 逻辑值（true 或 false）
  • 数组（在方括号中）
  • 对象（在花括号中）
  • null

JSON 解析器

提示：eval() 函数可编译并执行任何 JavaScript 代码。这隐藏了一个潜在的安全问题。

使用 JSON 解析器将 JSON 转换为 JavaScript 对象是更安全的做法。JSON 解析器只能识别 JSON 文本，而不会编译脚本。

在浏览器中，这提供了原生的 JSON 支持，而且 JSON 解析器的速度更快。

较新的浏览器和最新的 ECMAScript (JavaScript) 标准中均包含了原生的对 JSON 的支持。

mysql> grant all privileges on *.* to 'skuaka'@'%' identified by '你想设置的密码' with grant option;

  • all privileges：表示将所有权限授予给用户。也可指定具体的权限，如：SELECT、CREATE、DROP等。
  • on：表示这些权限对哪些数据库和表生效，格式：数据库名.表名，这里写*.*表示所有数据库，所有表。如果我要指定将权限应用到test库的user表中，可以这么写：test.user
  • to：将权限授予哪个用户。格式：”用户名”@”登录IP或域名”。%表示没有限制，在任何主机都可以登录。比如：”skuaka”@”192.168.0.%”，表示yangxin这个用户只能在192.168.0IP段登录。
  • identified by：指定用户的登录密码
  • with grant option：表示允许用户将自己的权限授权给其它用户

对用户做了权限变更之后，一定记得重新加载一下权限，将权限信息从内存中写入数据库。
