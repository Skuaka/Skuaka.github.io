access-list命令

（1）标准访问列表

access-list access-list-number { permit | deny } {source [source-wildcard] | any}

命令解释如下。

access-list：访问列表命令。

access-list-number：访问列表号码，值为1~99.

permit：允许。

deny：拒绝。

source：源IP地址。

source-wildcard：源IP地址的通配符。

（2）扩展访问列表

access-list access-list-number { permit | deny } { protocol \ protocol-keyword } { source [ source-wildcard ] | any } { destination destination-wildcard } | any }[protocol-specific options][log]

命令解释如下。

access-list-number：访问列表号码，值为100~199.

protocol \ protocol-keyword：可使用的协议，包括IP、ICMP、IGRP、EIGRP、OSPF等。

destination destination-wild：目的IP地址，格式与源IP地址相同。

protocol-specific options：协议制定的选项。

log：记录有关数据报进入访问列表的信息。

rip 跳数

offset-list acc号 in/out 跳数增加值

DM-VPN

动态多点VPN、Cisco私有

IPSEC VPN 存在的问题

  1. 不支持多播
  2. 复杂的拓扑

DM-VPN 的术语

  • M-GRE（多点隧道）一点对多点
  • 物理 IP 地址（物理接口）/逻辑 IP 地址（tunnel 私有 IP）
  • NHRP（下一跳解析协议、总部和分部都启用）
    NHS（下一跳服务器、总部）

DM-VPN 的三个阶段

阶段一

  • M-GRE：EIGRP
  • 分支和分支要通过总部才能相互通讯

阶段二

  • M-GRE：EIGRP
  • 分支和分支可以直接通讯

阶段三

  • 单云双中心（两个 HUB 同时工作互为备份，还是依赖 DM-VPN 阶段2）

DM-VPN 的基础配置流程

步骤1：完成 HUB 上 M-GRE 的配置

步骤2：完成 spoke 上的 M-GRE 配置

步骤3：完成 HUB 上隧道 NHRP 的配置

步骤4：完成 spoke 上 NHRP 的配置

步骤5：隧道逻辑 IP 地址参与 EIGRP

步骤6：完成 IPSEC 的配置并且套用到隧道上

杂：

查看 HUB 的分支信息：show ip nhrp

关闭 EIGRP 水平分割：no ip split-horizon eigrp 1

开启 NHRP 捷径：ip nhrp shortcat

区别1：两个 HUB 需要互做 NHRP 的映射。

CCIE 多播（multicast）

多播（multicast）

基础知识

数据的传递方式

  • 单播（unicast）

    一对一通讯，数据接收者已知。

  • 广播（boradcast）

    一对所有通讯，数据接收者未知。

  • 多播（mulicast）

    一对多通讯，数据接收者未知。

路由器和交换机对不同类型数据的处理方式

路由器

单播包：路由或丢弃

广播包：丢弃或本地存在感兴趣的进程进行处理

多播包：丢弃或本地存在感兴趣的进程进行处理

交换机

单播帧：目的MAC地址交换机是否已知，如果已知则交换，如果未知则泛洪

广播帧：泛洪

多播帧：泛洪

IGMP 互联网组管理协议

路由器及终端运行，帮助路由器发现直连子网中的多播接收者。

组（group）：所有对同一多播流感兴趣的成员组成了多播组。

组成员（group member）：组中的多播流接收者。

版本：IGMPv1、IGMPv2、IGMPv3

IGMPv1

  • 报告消息（report）：始发于终端，用于通告子网中的路由器存在特定组组成员。

    发送终端的IP 和MAC以及组信息。

  • 查询消息（request）：始发于路由器，用于查询直连子网中是否存在组成员。

    通常是路由器接收到多播流后触发。

一旦发现组成员，路由器就创造一个180s 的老化计时器，该计时器通过 report 消息进行维护。

IGMPv1的缺陷

  1. 响应时间慢

    组中没有成员时，路由器需要 180s 才能发现

  2. 存在多查询者

  3. 存在多转发者

IGMPv2

  • 报告消息（report）：和 v1 相同

  • 查询消息（request）：和 v1 相同

  • 离开消息（leave）：始发于终端，终端PC 不再是组成员时发送，用于通告路由器有组成员离开。

    收到离开消息时，立刻发送SQ，如下。

  • 特定查询消息（SQ）：始发于路由器，发送出去后设定的老化时间为 3s 。如果 3s 内收到 report ，说明发送离开消息的组成员并不是组中最后一个人；如果收不到任何消息，那么说明组成员不存在。

IGMPv2 查询者竞选机制：多个查询者通过比较查询消息源 ip 来确认谁最终负责查询，源 ip 越小越优。

多转发者缺陷 IGMPv2 无法克服。

多播地址

多播 IP 和多播 MAC 都不是用来定位接收者所在的网络和物理位置。

它们用于标识服务和应用。

RIPv2 224.0.0.9

IPv4 D类 224.x.x.x 239.x.x.x

多播 mac

设备根据多播的 IP 自动计算而来。

多播的MAC地址可能会重复，多播的 IP 后 23bit 一致则会重复。

多播路由器协议

是运行在路由器之间，负责建立多播路由路径，用于传递多播数据。

多播路由协议分类

  • 距离矢量：DVMRP 距离矢量多播路由协议（淘汰了）
  • 链路状态：MOSPF 多播的OSPF（非主流）
  • 距离向量：CBT 基于核心的树（非主流）
  • 独立于其他：PIM 协议独立多播（Cisco、主流）

PIM 的两个版本

PIMv1：用的是 IGMPv1，但是现在也用 IGMPv2

PIMv2：用的是 IGMPv2

PIM 的三种模式

  • 密集模式（dense mode）

    设备间带宽资源紧张，子网成员大部分是组成员的时候，建议使用密集模式。

    密集模式默认采用 IGMPv1。

  • 稀疏模式（sparse mode）

    设备间带宽资源依旧紧张，子网成员只有少数是组成员，建议使用稀疏模式。

    稀疏模式采用 IGMPv2。

  • 混合模式（sparse-dense mode）

    该模式下路由器先启用稀疏模式，稀疏模式失效后再用密集模式。

PIM 的密集模式详解

消息

  • 加入消息（join）：下游路由器通过加入消息将直连网络加入多播转发路径。
  • 修剪消息（prune）：下游路由器通过修剪消息，将原本隶属于多播路径的直连网络从多播路径中删除。
  • 接枝消息（graft）：下游路由器通过该消息请求将原本被拆除的多播路径再恢复过来。
  • 接枝确认消息（graft-ack）：上游路由器通过该消息告诉下游，直连路径恢复成多播转发路径。
  • assert消息（assert）：连接子网的路由器通过该消息，竞选出最终的转发者。

密集模式工作过程

扩散、剪枝。

PIM 的特殊机制

  • PIM 密集模式的 DR

    接口优先级+接口ip地址，构成竞选参数，越大越优。

    PIM密集模式早期使用IGMPv1，IGMPv1会出现多查询者问题，于是就利用DR的竞选，来帮助IGMPv1确认谁负责查询。

    PIM密集模式现在使用的是IGMPv2，DR就没用了。

  • PIM 消息中的 assert 作用

    通过竞选选出一个人负责转发数据。

    assert 消息，携带了三个竞选参数：

    • 当前路由器启用的底层协议管理距离，越小越优。
    • 当前路由器抵达多播源底层路由协议的开销，越小越优。
    • 当前路由器连接子网的接口 ip，越大越优。

PIM 密集模式的配置流程

  • 步骤1：确认参与多播的设备及接口

    所有需要接收多播流的接口和发送多播流的接口及连接组成员的接口，都要参与多播路由协议的启用。

  • 步骤2：路由器接口模拟成多播组成员

    注意：该命令可以反复录入，一个接口扮演多个组成员

  • 步骤3：路由器开启多播路由功能，接口启用多播密集模式

  • 步骤4：校验查看

    校验路由器是否发现了组成员：show ip igmp group

    校验路由器是否形成了PIM 邻居关系：show ip pim neighbor

    校验路由器多播路由表：show ip mroute

多播路由表

父条目与子条目

  • 父条目

    父条目起提醒注释的作用，告知当前路由器和管理员，路由器可能需要为某个多播组传递流量。

    路由器收到 report 消息便会产生父条目，路由器收到多播流也会产生父条目。

  • 子条目

    子条目是实际负责多播流量传递工作。

    子条目是路由器根据计算得来。

多播路由条目的样式

（多播源，组地址） 存活计时器/老化计时器

Incoming-interface（上游接口）：接口名字 状态/模式

Outgoing-interface（上游接口）：接口名字 状态/模式

父条目多播源部分几乎都是*号，代表源未知。

子条目多播源部分一定是某个 IPv4 单播地址。

多播防环机制

RPF：反向路径转发：启用多播的接口默认开启，用于避免环路的出现。

所有接收多播流的接口，必须通过 RPF 检测，才可以成为有效的上游接口。

当前接口必须是抵达多播源下一跳地址的本地出口，才通过检测。

用来查看rpf检测是否通过：show ip rpf <intf-id>

多播树

多播树其实就是多播转发数据的路径。

密集模式的树

最短路径树，也称源树。

当前路由器以多播源为根，计算抵达多播源的最短路径，每个路径对应到多播路由表中的一个条目。

源树缺陷
  • 容易产生大量的多播子条目。
  • 树构建的早期多播流被扩散到了不必要的设备上

PIM 的稀疏模式详解

消息

join、prune、graft、graft-ack、assert（和密集模式相同）

注册消息（register）

注册停止消息（register-stop）

稀疏模式中的集合点：RP

RP 是管理员指定的，是多股多播流的交汇处。

稀疏模式的工作原理

上游注册，下游加入，RP 整合。

！！注意

RP 以下的设备，以 RP 为根，计算最短路径。

RP 自身以源为根，计算最短路径。

稀疏模式是一个双树结构，RP 和源之间维护的是源树，下游设备与 RP 之间维护的是共享树。

共享树切换成源树

稀疏模式下，每台支持多播路由协议的路由器在接收超过4个字节的多播流后，就会尝试建立源树。

RP 的获取方式

  • 手动指定：ip pim rp-address <RP地址>

  • RP 的竞选

    1. 代理 rp（auto-rp）

      C-RP：RP候选者

      RP-Agent：RP代理、C-RP 信息收集，裁决，分享结果

      RP：RP

    2. BSR：boot strap router：自举路由器

      C-BSR：BSR 候选人

      BSR：信息收集，信息分享

      C-RP、RP

      BSR 的竞选过程：BSR 标识越大越优。

      RP

      C-RP 有三个竞选参数：优先级（小优）、hash 长度（长优）、C-RP 标识（大优）

PIM 的混合模式

让设备启用混合模式：if)#ip pim sparse-dense-mode

混合模式倒退回密集模式：一定是 RP 彻底无法正常工作

Router(config)#no ip pim dm-fallback //不允许混合模式倒退回密集模式

auto-rp 的问题和解决方案

224.0.1.39 与 224.0.1.40

C-RP 将竞选信息发向 224.0.1.39

RP-Agent 将结果发向 224.0.1.40

所有设备为了正确的获取多播 RP 信息，默认会接收发向这两个地址的多播流。

auto-rp 结果通告信息的跳数

通告结果是一跳，如果拓扑是水平的多台设备组成，那么就会出现有设备无法得知 RP 信息。

PIM 稀疏模式 DR 的作用——没记全！！！

路由器连接子网的部分，DR 负责发送 join 信息给 RP，路由器连接多播源

网关冗余对多播传递的影响

HSRP 热备份路由协议，起在路由器接口下。

子网通常被要求单播数据流和多播数据流分行。

把当前设备设置成RP-Agent：ip pim send-rp-discovery lo0 sc 255

配置 RP 侦听

RP-Agent 公布 rp 结果的消息只有一跳，所以只能传给直连设备，该跳数可以修改。

路由器默认不传递 RP-Agent RP 结果，还必须配置 RP 侦听，帮助路由器透传。

ip pim autorp listener

CCIE 交换机基础

交换机基础

泛洪：除接收端口外，所有端口都向外去发。

mac地址表=cam表

广播域：所能传播的最大范围

冲突域：发生冲突之后所影响的范围

CSMA/CD：载波帧听多链路访问

交换机指示灯：

绿色常亮：接口两端没有问题，但是当前没有数据从中通过

绿色不停地闪：正在传输数据

橘红色慢闪：端口没问题,对端线路或者对端的端口有问题

橘红色快闪：端口和对端连接没有问题，正在协商参数建立

黄色闪动：物理接口有问题，尝试重启。

不闪： 物理接口有问题，无法通信

vlan（虚拟局域网）

作用：分隔广播域

所有接口默认属于vlan1（默认vlan，不可删除）。

vlan ID范围和解释

\1. 0和4095 仅仅限于系统使用 用户不能查看和使用

\2. 1 正常 cisco默认的vlan,用户可以使用但是不能删除

\3. 2–1001 正常 用于以太网的VLAN，用户可以创建。使用。删除。

\4. 1002–1005 正常 用于FDDI和令牌环的CISCO默认VLAN，用户不能删除

\5. 1006–1024 保留 仅仅限于系统使用。用户不能查看和使用。

\6. 1025–4094 扩展 仅仅用于以太网VLAN

CCIE 跨域的多播

跨域的多播

源和组成员在不同的AS，需要传递多播数据流。

跨域多播的问题和挑战

组成员无法发现源、源无法发现组成员、多播树无法建立。

MSDP 多播源发现协议

MSDP 是配置在不同 AS RP 上的，它会建立基于 TCP 的对等体关系，利用自身的 SA（源关联）消息共享”源地址信息、多播组地址信息、RP 信息”给对端 RP。

MSDP 配置流程

步骤1：RP上配置新的 loopback 接口，并且宣告进 BGP。

步骤2：RP上完成 MSDP 的配置

排错手段

ip 路由疑难解析

多播排错

ping 多播组地址 不同

  1. 确认发现组成员

    show ip igmp group

    查看组地址、本地接口、报告者 是否出现

    错点

    • 没有模拟成组成员，缺乏接口下：ip igmp join-group <a.b.c.d>
    • 路由器接口没有加入PIM，缺乏接口下:ip pim <x>
    • 连通性问题

  2. 确认稀疏模式获取 RP 信息

    show ip pim rp mapping 查看 RP 映射，查看 RP 的结果

    show ip pim rp 查看多播组和 RP 的关联

    错点

    • 没有通过静态 RP 指明 RP 地址
    • BSR 的 C-BSR 或 C-RP 配置问题
    • auto-rp 的配置问题
    • 缺乏auto-rp 侦听的配置
    • 设备间缺乏必要的PIM 邻居：show ip pim neighbor
      • 设备缺乏多播路由功能和接口忘记加入PIM
      • 接口忘记激活、错误的安全策略挡住了PIM消息
    • 设备无法正常抵达 RP 地址，所以不显示获取的 RP 消息
      • 路由层面问题、忘记宣告、忘记重分发、错误的过滤

  3. RPF 检测未通过

    • 缺乏必要的更新源路由
    • 忘记宣告、忘记重分发、错误的路由更新过滤

  4. 是否是跨域多播、缺乏必要的配置和连通性

    • 缺乏必要的 MSDP 配置和 MSDP 对等体关系

      show run | sec ip msdp

      show ip msdp summary

      MSDP peer 地址不可达

      MSDP 链接源和起源 ID 设置错误

    • RPF 检测不通过

  5. 错误的安全策略

CCIE 重分发

重分发

将不同路由协议转化

1直连进路由选择协议

redistribute connect metric 1

2其他路由选择协议进rip

router rip

router下 redistribute eigrp 100 metric 1值(默认0)

3其他路由选择协议进eigrp

router eigrp 100

redistribute rip metric 10000带宽 100延迟 255可靠性 1负载 1500MTU值（默认为以上数字，一般不改）

4其他进ospf

router os 1

redistribute eigrp 100进程号 subnets（子集，不加则为主类网络）

默认OE2

将oe2改成oe1:

re ei 100 subnets metric-type 1 (+metric 值）

重分发流量控制

acl

奇acc 1 permit 172.16.1.0 0.0.254.255

偶acc 1 permit 172.16.0.0 0.0.254.255

将acl抓去到的流量给route-map

route-map +名字 +permit/deny

优先级10(默认优先级)

match ip address +acl编号

route-map也默认deny所有流量

如需放行所有流量 route-map 名字 permit +下一个优先级20(一般为10的叠加)

re rip router-map ccie metric ……
