access-list命令

（1）标准访问列表

access-list access-list-number { permit | deny } {source [source-wildcard] | any}

命令解释如下。

access-list：访问列表命令。

access-list-number：访问列表号码，值为1~99.

permit：允许。

deny：拒绝。

source：源IP地址。

source-wildcard：源IP地址的通配符。

（2）扩展访问列表

access-list access-list-number { permit | deny } { protocol \ protocol-keyword } { source [ source-wildcard ] | any } { destination destination-wildcard } | any }[protocol-specific options][log]

命令解释如下。

access-list-number：访问列表号码，值为100~199.

protocol \ protocol-keyword：可使用的协议，包括IP、ICMP、IGRP、EIGRP、OSPF等。

destination destination-wild：目的IP地址，格式与源IP地址相同。

protocol-specific options：协议制定的选项。

log：记录有关数据报进入访问列表的信息。

rip 跳数

offset-list acc号 in/out 跳数增加值

MPLS详解

multiple protocol label switch（多协议标签交换）

MPLS作用

MPLS是一种数据交换机制，基于标签库和数据携带的标签。

进行数据转发，同时支持多种协议。

MPLS在实际工作中，能实现如下两个基本功能：

  1. 数据包的高速转发
  2. 减少参与BGP的设备数量、减少对等体关系、减少不必要的路由学习

MPLS术语

label 标签

总长 32bit 由如下几个部分组成：

前20bit 是标签取值范围

第21bit 是栈底位，栈底位为1则表明该标签已经是最后一个

第22~24bit 是体验位，可以将标签对应的数据分成8个级别，给予不同的传输策略。

最后8bit 是TTL，限制携带标签的数据传递范围。

MPLS 模式

帧模式：frame mode（主流模式），标签被插在帧头的后面，包头前面。

信令模式：cell mode 标签被插在信令头后面。

LSR：标签交换路由器

启用了MPLS的路由器统称LSR。

入站 LSR （inbound LSR）：接收到 IP 数据包，压入标签，传递给 MPLS 网络。

链路中 LSR （link LSR）：接收到携带标签的数据，基于标签进行数据转发。

出站 LSR （outbound LSR）：负责将标签移除，并且变回 IP 数据报文。

压入、交换、移除、弹出

压入：入站LSR将标签压入数据。

交换：链路中LSR基于标签库，进行标签的置换、数据的传递。

移除：出站LSR将数据携带的所有标签全部删除。

弹出：出站LSR将数据携带的栈顶标签弹出，其他标签保留。

LSP：链路标签路径

是MPLS设备传递数据所需的路径。（即路由）

LSP是通过标签的关联产生的。

标签分配、分发、关联、维护协议

TDP：标记分发协议、思科私有

LDP：标签分发协议、公有公开

LDP详解

LDP作用

  1. 利用 hello 消息建立 LDP 邻接关系。
  2. 基于CEF给本地网络分配本地标签。
  3. 将网络对应的标签分发共享给邻居
  4. 基于底层协议完成本地标签与远程共享标签的关联，最终形成LSP。
  5. 负责监控LSP的有效性，并且维护它。

LDP邻接关系的建立

LDP 邻居的发现，通过hello消息，基于UDP，目标端口646。

LDP 利用TCP，目标端口646完成邻接关系建立。

LDP可以设定RID，RID扮演更新源，更新源会成为对端TCP的目标接收地址，所以更新源必须可达。

LDP标签的分配

除BGP路由，LDP可以给本地直连网络以及非直连网络分配标签。

LDP标签的共享

两种形式：下游主动、上游主动。

实际上标签的分享是给所有的邻居，不遵循所谓的上下游。

LDP标签的关联

LDP永远将路由路径下一跳共享的标签与对应网络的本地标签关联。

boundaddress：绑定地址，参与MPLS设备的IP会成为绑定地址和，会跟着本地标签一起共享给LDP邻居，邻居根据底层协议下一跳地址和绑定地址的判断来确认哪个LDP邻居是下一跳，再绑定。

LDP的特殊标签

  • UNTAGE：未标记

出现原因：底层是ospf,本地网络无法找到对应的远程标签，LDP给网络分配untage。

处理方式：目标网络远程标签如果是untage，那么LSR会将所有标签移除，向外传递。

  • POP：倒数第二跳标签

出现原因：当前路由器给本地直连网络分配的本地标签就是POP。

处理方式：如果出站标签是POP，那么LSR 会将栈顶标签弹出。

意义：提高数据转发效率，减少一次标签库查找。

LDP的两张表

LIB：标签信息库

目标网络本地标签和学习到的远程标签存储在这里。（未关联）

LFIB：标签转发信息库

标签关联后的结果 即 LSP 存储在这里。

MPLS的配置流程

步骤1：开启 CEF（默认开启，一般情况不需要此命令）

ip cef

步骤2：设置标签分配、分发关联协议（默认是LDP）

mpls label protocol <ldp/tdp>

步骤3：设置标签取值范围（通常不设置）

mpls label range <xx-yy>

步骤4：设置 LDP 的 RID 即设置更新源（不设置则和OSPF一样自动选取RID）

mpls ldp router-id loopback x [force] //RID 必须可达、force 强制改变 RID

步骤5：接口下启用MPLS

mpls ip

CCIE DM-VPN

DM-VPN

动态多点VPN、Cisco私有

IPSEC VPN 存在的问题

  1. 不支持多播
  2. 复杂的拓扑

DM-VPN 的术语

  • M-GRE（多点隧道）一点对多点
  • 物理 IP 地址（物理接口）/逻辑 IP 地址（tunnel 私有 IP）
  • NHRP（下一跳解析协议、总部和分部都启用）
    NHS（下一跳服务器、总部）

DM-VPN 的三个阶段

阶段一

  • M-GRE：EIGRP
  • 分支和分支要通过总部才能相互通讯

阶段二

  • M-GRE：EIGRP
  • 分支和分支可以直接通讯

阶段三

  • 单云双中心（两个 HUB 同时工作互为备份，还是依赖 DM-VPN 阶段2）

DM-VPN 的基础配置流程

步骤1：完成 HUB 上 M-GRE 的配置

步骤2：完成 spoke 上的 M-GRE 配置

步骤3：完成 HUB 上隧道 NHRP 的配置

步骤4：完成 spoke 上 NHRP 的配置

步骤5：隧道逻辑 IP 地址参与 EIGRP

步骤6：完成 IPSEC 的配置并且套用到隧道上

杂：

查看 HUB 的分支信息：show ip nhrp

关闭 EIGRP 水平分割：no ip split-horizon eigrp 1

开启 NHRP 捷径：ip nhrp shortcat

多播（multicast）

基础知识

数据的传递方式

  • 单播（unicast）

    一对一通讯，数据接收者已知。

  • 广播（boradcast）

    一对所有通讯，数据接收者未知。

  • 多播（mulicast）

    一对多通讯，数据接收者未知。

路由器和交换机对不同类型数据的处理方式

路由器

单播包：路由或丢弃

广播包：丢弃或本地存在感兴趣的进程进行处理

多播包：丢弃或本地存在感兴趣的进程进行处理

交换机

单播帧：目的MAC地址交换机是否已知，如果已知则交换，如果未知则泛洪

广播帧：泛洪

多播帧：泛洪

IGMP 互联网组管理协议

路由器及终端运行，帮助路由器发现直连子网中的多播接收者。

组（group）：所有对同一多播流感兴趣的成员组成了多播组。

组成员（group member）：组中的多播流接收者。

版本：IGMPv1、IGMPv2、IGMPv3

IGMPv1

  • 报告消息（report）：始发于终端，用于通告子网中的路由器存在特定组组成员。

    发送终端的IP 和MAC以及组信息。

  • 查询消息（request）：始发于路由器，用于查询直连子网中是否存在组成员。

    通常是路由器接收到多播流后触发。

一旦发现组成员，路由器就创造一个180s 的老化计时器，该计时器通过 report 消息进行维护。

IGMPv1的缺陷

  1. 响应时间慢

    组中没有成员时，路由器需要 180s 才能发现

  2. 存在多查询者

  3. 存在多转发者

IGMPv2

  • 报告消息（report）：和 v1 相同

  • 查询消息（request）：和 v1 相同

  • 离开消息（leave）：始发于终端，终端PC 不再是组成员时发送，用于通告路由器有组成员离开。

    收到离开消息时，立刻发送SQ，如下。

  • 特定查询消息（SQ）：始发于路由器，发送出去后设定的老化时间为 3s 。如果 3s 内收到 report ，说明发送离开消息的组成员并不是组中最后一个人；如果收不到任何消息，那么说明组成员不存在。

IGMPv2 查询者竞选机制：多个查询者通过比较查询消息源 ip 来确认谁最终负责查询，源 ip 越小越优。

多转发者缺陷 IGMPv2 无法克服。

多播路由器协议

是运行在路由器之间，负责建立多播路由路径，用于传递多播数据。

多播路由协议分类

  • 距离矢量：DVMRP 距离矢量多播路由协议（淘汰了）
  • 链路状态：MOSPF 多播的OSPF（非主流）
  • 距离向量：CBT 基于核心的树（非主流）
  • 独立于其他：PIM 协议独立多播（Cisco、主流）

PIM 的两个版本

PIMv1：用的是 IGMPv1，但是现在也用 IGMPv2

PIMv2：用的是 IGMPv2

PIM 的三种模式

  • 密集模式（dense mode）

    设备间带宽资源紧张，子网成员大部分是组成员的时候，建议使用密集模式。

    密集模式默认采用 IGMPv1。

  • 稀疏模式（sparse mode）

    设备间带宽资源依旧紧张，子网成员只有少数是组成员，建议使用稀疏模式。

    稀疏模式采用 IGMPv2。

  • 混合模式（sparse-dense mode）

    该模式下路由器先启用稀疏模式，稀疏模式失效后再用密集模式。

PIM 的密集模式详解

消息

  • 加入消息（join）：下游路由器通过加入消息将直连网络加入多播转发路径。
  • 修剪消息（prune）：下游路由器通过修剪消息，将原本隶属于多播路径的直连网络从多播路径中删除。
  • 接枝消息（graft）：下游路由器通过该消息请求将原本被拆除的多播路径再恢复过来。
  • 接枝确认消息（graft-ack）：上游路由器通过该消息告诉下游，直连路径恢复成多播转发路径。
  • assert消息（assert）：连接子网的路由器通过该消息，竞选出最终的转发者。

密集模式工作过程

扩散、剪枝。

PIM 的特殊机制

  • PIM 密集模式的 DR

    接口优先级+接口ip地址，构成竞选参数，越大越优。

    PIM密集模式早期使用IGMPv1，IGMPv1会出现多查询者问题，于是就利用DR的竞选，来帮助IGMPv1确认谁负责查询。

    PIM密集模式现在使用的是IGMPv2，DR就没用了。

  • PIM 消息中的 assert 作用

    通过竞选选出一个人负责转发数据。

    assert 消息，携带了三个竞选参数：

    • 当前路由器启用的底层协议管理距离，越小越优。
    • 当前路由器抵达多播源底层路由协议的开销，越小越优。
    • 当前路由器连接子网的接口 ip，越大越优。

PIM 密集模式的配置流程

  • 步骤1：确认参与多播的设备及接口

    所有需要接收多播流的接口和发送多播流的接口及连接组成员的接口，都要参与多播路由协议的启用。

  • 步骤2：路由器接口模拟成多播组成员

    注意：该命令可以反复录入，一个接口扮演多个组成员

  • 步骤3：路由器开启多播路由功能，接口启用多播密集模式

  • 步骤4：校验查看

    校验路由器是否发现了组成员：show ip igmp group

    校验路由器是否形成了PIM 邻居关系：show ip pim neighbor

    校验路由器多播路由表：show ip mroute

多播路由表

父条目与子条目

  • 父条目

    父条目起提醒注释的作用，告知当前路由器和管理员，路由器可能需要为某个多播组传递流量。

    路由器收到 report 消息便会产生父条目，路由器收到多播流也会产生父条目。

  • 子条目

    子条目是实际负责多播流量传递工作。

    子条目是路由器根据计算得来。

多播路由条目的样式

（多播源，组地址） 存活计时器/老化计时器

Incoming-interface（上游接口）：接口名字 状态/模式

Outgoing-interface（上游接口）：接口名字 状态/模式

父条目多播源部分几乎都是*号，代表源未知。

子条目多播源部分一定是某个 IPv4 单播地址。

多播防环机制

RPF：反向路径转发：启用多播的接口默认开启，用于避免环路的出现。

所有接收多播流的接口，必须通过 RPF 检测，才可以成为有效的上游接口。

当前接口必须是抵达多播源下一跳地址的本地出口，才通过检测。

用来查看rpf检测是否通过：show ip rpf <intf-id>

多播树

多播树其实就是多播转发数据的路径。

密集模式的树

最短路径树，也称源树。

当前路由器以多播源为根，计算抵达多播源的最短路径，每个路径对应到多播路由表中的一个条目。

源树缺陷
  • 容易产生大量的多播子条目。
  • 树构建的早期多播流被扩散到了不必要的设备上

IS-IS（中间系统到中间系统）

数据传递方式

  • 面向连接 tcp/ip

  • 非面向连接 clns/clnp 非连接的网络服务(负责控制数据传递过程)/非连接的网络协议(网络地址标识)

    最后，ip获得广泛使用，但是CLNS的理念也得到认可，因此保留下来，叫做UDP。

IS-IS术语

  • IS：中间系统 即 路由器router

  • ES：终端系统 （但是现在已经没有使用clnp标识的设备了）

  • clnp：标识

    NET：网络实体标识（网络+主机）128bit

    clnp：clnp地址 NSAP网络服务应用标识

clnp地址解释

0101 01/01 域/区域 （区域包含在域内）这里为两个十六进制数

0000.0000.0001 mac地址

00 服务号（作用类似于端口号）

  • IS-IS：Dis

OSPF七种LSA类型

  • LSA1（Router LSA）

    起源：区域内每台设备都能产生LSA1

    内容：节点信息{RID network cost}

    扩散：不允许穿越ABR，也就是区域内扩散。

    路由：区域内路由

  • LSA2（Network LSA）

    起源：只有DR可以产生

    内容：拓扑信息{节点的直连情况，谁是DR}

    扩散：区域内

  • LSA3（Network summary LSA）

    起源：ABR

    内容：区域网络信息

    扩散：整个OSPF AS

    路由：区域间路由

  • LSA4（ASBR summary LSA）

    起源：ASBR所在域的ABR

    内容：ASBR的RID

    扩散：整个OSPF

    路由：外部{OE1，OE2}

  • LSA5（Autonomous system external LSA）

    起源：ASBR

    内容：外部路由

    扩散：整个OSPF

    路由：外部{OE1，OE2}

CCIE STP生成树协议

STP生成树协议 7.27

一个广播域中选举一个生成树。

广播风暴：绝不停止。

重复帧。

mac地址表的不稳定。

1一个网络中选取一个根桥

2根桥上所有端口都为指定端口

3非根桥上选取一个根端口

4每一段选举一个指定端口

5非指定端口为阻塞端口

*粗体为生成树定义的交换机接口角色

根端口：距离根桥距离最短的端口。开销值一样的话就选最小的发送者id。只在非桥根上选举，并且只选一个。

指定端口：可以执行数据的转发。一台交换机上可以有多个指定端口。根桥上所有端口都是指定端口。

阻塞端口：逻辑上的阻塞，不执行数据的转发，但会持续侦听bpdu信息。

BPDU(桥协议数据单元)只允许根桥产生

root id

优先级：(默认32768，越小越优，更改 优先值时必须为4096的倍数)。

背板mac：标识一台交换机，背板mac小的为优。

bridge id

优先级+背板mac：和根id一样

cost

开销值

port id

优先级(默认128，小优，更改时必须为64的倍数)

端口mac：发送者的端口mac

PVST：每vlan生成树，在优先级插入vlan的信息。优先级+vlan编号。

端口状态

Blocking（阻塞状态）：此时，二层端口为非指定端口，也不会参与数据帧的转发。该端口通过接收BPDU来判断根交换机的位置和根ID，以及在STP拓扑收敛结束之后，各交换机端口应该处于什么状态，在默认情况下，端口会在这种状态下停留20秒钟时间。

Listening（侦听状态）：生成树此时已经根据交换机所接收到的BPDU而判断出了这个端口应该参与数据帧的转发。于是交换机端口就将不再满足于接收BPDU，而同时也开始发送自己的BPDU，并以此通告邻接的交换机该端口会在活动拓扑中参与转发数据帧的工作。在默认情况下，该端口会在这种状态下停留15秒钟的时间。

Learning(学习状态)：这个二层端口准备参与数据帧的转发，并开始填写MAC表。在默认情况下，端口会在这种状态下停留15秒钟时间。

Forwarding（转发状态）：这个二层端口已经成为了活动拓扑的一个组成部分，它会转发数据帧，并同时收发BPDU。

Disabled（禁用状态）：这个二层端口不会参与生成树，也不会转发。
